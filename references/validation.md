# OmniBox 脚本修改后的校验规范

本文档用于统一 OmniBox 脚本修改后的最小校验要求。

目标：
- 避免“代码改了但脚本根本跑不起来”
- 至少保证语法/编译层面正常
- 降低把明显错误脚本交付给用户的概率

---

## 一、强制规则

### 1. 每次改完脚本后必须做一次静态检查
无论改动大小，只要脚本内容发生变化，就必须做至少一次静态校验。

包括但不限于：
- 修 bug
- 加功能
- 改环境变量
- 改返回结构
- 改排序逻辑
- 改刮削 / 弹幕 / 播放历史

### 2. 最低要求：至少语法/编译正常
不能只“看起来像对了”，必须至少验证：
- JavaScript：语法可通过
- Python：语法可通过

---

## 二、JavaScript 校验方式

### 最低要求
使用：
```bash
node --check /path/to/script.js
```

### 通过标准
- 命令退出码为 0
- 没有语法错误

### 说明
这不能保证运行时逻辑完全正确，但至少能避免：
- 括号没闭合
- 语法拼错
- 结构损坏
- 直接无法加载

---

## 三、Python 校验方式

### 最低要求
使用：
```bash
python3 -m py_compile /path/to/script.py
```

### 通过标准
- 命令退出码为 0
- 没有语法错误

### 说明
这不能保证逻辑完全正确，但至少保证脚本能被 Python 正常解析。

---

## 四、推荐补充检查

在最低语法检查之外，若条件允许，建议再做：

### 1. 关键变量存在性检查
例如：
- 是否误删 `SOURCE_NAMES_CONFIG`
- 是否误删 `DRIVE_TYPE_CONFIG`
- 是否误删 `runner.run(...)`
- 是否误删 `module.exports`

### 2. 关键函数位置检查
例如：
- 排序逻辑是否真的插在 `detail()` 而不是 `search()`
- 新增逻辑是否插入到返回值之前

### 3. 版本号检查
确认：
- 改完后 `@version` 已升级
- 升级幅度符合 patch / minor / major 规则

---

## 五、交付前的默认检查清单

以后每次修改 OmniBox 脚本，默认至少检查：

### JavaScript 脚本
- [ ] `@version` 是否已升级
- [ ] `node --check` 是否通过
- [ ] 关键环境变量是否还存在
- [ ] 新逻辑是否插在正确函数中

### Python 脚本
- [ ] `@version` 是否已升级
- [ ] `python3 -m py_compile` 是否通过
- [ ] 关键环境变量是否还存在
- [ ] 新逻辑是否插在正确函数中

---

## 六、默认执行策略（强制）

以后只要我帮你改 OmniBox 脚本：
1. 默认修改代码
2. 默认升级 `@version`
3. 默认做至少一次静态检查
4. 默认在交付说明里写明“已通过静态检查”或“检查方式”

如果没有通过静态检查，就不应把脚本当成“完成版”交付。
